Executive Summary
This paper introduces a novel approach to Android malware-family classification that initializes Transformer
embeddings using global Markov-chain statistics from API call transitions. The authors evaluate the impact
of this “Markov initialization” on both classification accuracy and explainability (via attention “faithfulness”
in a deletion test). On a 9,337-sample dynamic API-call dataset (8 malware families from Droidmon), the
Markov-initialized Transformer achieves lower accuracy (≈0.927 vs 0.940 for a plain Transformer) but 
greater attention faithfulness (e.g. +10.1% faithfulness gap at k=20) compared to random initialization.
The study highlights an accuracy–explainability tradeoff, arguing that improved interpretability may be
worth a small accuracy cost in high-stakes settings. 
Overall, the paper is clearly written and introduces an interesting idea. The approach appears technically
sound, and the evaluation on a real malware dataset is valuable. However , there are significant concerns
that should be addressed before publication:
Reproducibility Details: Key methodological details are missing (e.g. full model architecture,
training hyperparameters, random seeds, SVD dimensions), and code is not provided. This hinders
reproducibility. The authors should follow established best practices (e.g. reproducibility checklists
) by listing all hyperparameters, number of runs, and sharing code. A clear Code Availability
statement and public repository (as recommended by Nature portfolio guidelines) would greatly
strengthen this work. 
Evaluation Scope: The experiments are limited to one dataset (Droidmon traces) and 3-fold CV.
Generalization to other benchmarks (e.g. Drebin, CICAndMal2017) or unseen families is not tested.
We recommend additional experiments or at least a more thorough discussion of generalizability.
Tables comparing this dataset to others (size, features, families) would clarify the context. For
example, including a table of related datasets and their characteristics (e.g. Drebin: 5560 apps, 149
families; CICMal2017: >2M samples, etc.) would be informative.
Statistical Analysis: No significance tests are reported. Differences in accuracy or faithfulness could
be due to chance given only 3 folds. We suggest applying appropriate tests (e.g. McNemar’s test for
paired accuracies) or bootstrapped confidence intervals. A table of performance metrics with p-
values or confidence intervals would clarify which improvements are statistically significant.
Literature and Context: Related work on explainable malware detection and Transformer-based
malware models should be cited and discussed. For example, Soi et al. (2024) use 1D-CNNs with
SHAP to explain Android malware API sequence classification, and Wu et al. (2021), Tanha &
Kafaie (2025), etc., apply XAI but focus on single APIs. This work goes further by encoding
transition patterns, which is novel. The authors should explicitly cite such studies to clarify how their
method differs. 
Clarity: A minor but confusing point is the use of “k” in two ways: as the spacing in the Markov
model (k∈[1,10]) and as the number of tokens masked (k=5,10,20). This overload can confuse

Major Issues
Reproducibility (Major) – Issue: The paper omits many critical details needed for replication. For
example, the Transformer architecture (number of layers/heads, embedding dimension), training
hyperparameters (learning rate, batch size, epochs, optimizer), dataset split procedure, and random
seeds are not fully specified. The description of the SVD embedding (e.g. how many singular vectors
are used) is unclear . Suggested Fix: Provide a complete Methods description or supplementary
material detailing all experimental settings. Share code and data processing scripts in a public
repository and include a “Code Availability” section (as advised by Nature’s guidelines). Follow the
ML reproducibility checklist: list all hyperparameter ranges, number of runs, and include instructions
to reproduce results. Expected Impact: This will allow reviewers and readers to verify and build on
the results, enhancing the paper’s credibility. 
Limited Dataset and Evaluation (Major) – Issue: Results are shown on only one dataset (Droidmon
API traces) with 8 malware families. It is unclear how well the approach works on other benchmarks
or on unseen malware families. The generalizability is therefore unknown. Suggested Fix: Test the
method on additional public datasets (e.g. Drebin, CICAndMal2017) or at least perform cross-dataset
validation. If new experiments are not feasible, the authors should more strongly qualify their claims
and discuss limitations (they already note the single-dataset issue in passing). Including a table
comparing the current dataset to others (size, number of classes, dynamic vs static) would
contextualize the work. Expected Impact: Broader evaluation or clearer limitations would
demonstrate whether the approach is robust or rather dataset-specific, improving the study’s
significance and trustworthiness.
Statistical Significance (Minor) – Issue: No statistical tests or confidence intervals are reported.
With only 3 CV folds, reported “mean ± std” might not capture significance. The observed accuracy
drop (94.0% to 92.7%) and faithfulness gains might be due to chance. Suggested Fix: Perform
appropriate statistical tests (e.g. McNemar’s test for paired accuracy or t-tests on metric
differences) to verify the differences. Present p-values or confidence intervals (perhaps in a table).
Alternatively, increase the number of repeats to get more reliable estimates (e.g. 5-fold CV or
multiple random splits). Expected Impact: This will clarify which performance changes are statistically
meaningful, lending rigor to the claims of improved interpretability vs dropped accuracy.
Explainability Claim (Major) – Issue: The paper claims “more interpretable” or “more faithful”
attention due to Markov init, but the evidence is based on a specific deletion test metric. While this is
sensible, the authors should ensure they do not overstate conclusions. For example, in one case
(GinMaster), the plain Transformer had a much higher faithfulness gap (51.9% vs 12.0%). This
suggests the Markov approach can hurt explainability when its priors misalign. Suggested Fix: Discuss

Literature Coverage (Minor) – Issue: The Related Work section mentions CNN/LSTM-based malware

cite any prior work that uses co-occurrence or Markov ideas for embeddings (aside from classic NLP
embeddings [12,13]). Suggested Fix: Cite and briefly discuss relevant work, such as Soi et al. (2024)
who use CNN+SHAP on API sequences, and recent GNN-based XAI approaches. Compare
and contrast these to the current work. Also clarify that, as claimed, embedding initialization via
Markov is novel (no obvious prior method is known). Expected Impact: This situates the contribution
in the broader context and shows awareness of related methods in explainable Android malware
detection.
Notation and Clarity (Minor) – Issue: The symbol k is used for two different meanings: the spacing
parameter in the Markov model (1≤k≤10) and the number of tokens masked in the deletion test
(k=5,10,20). This can confuse readers. Suggested Fix: Use different notation (e.g. k vs m) or explicitly
clarify in text/table captions. Also fix minor typos/formatting (e.g. missing leading “0.” in Table II
entries). Expected Impact: Improves readability; avoids confusion.
Minor Issues
Writing and Organization: Overall the paper is well-organized. A couple of sentences could be
rephrased for clarity (e.g. the description of the deletion test could be more intuitive). Proofread to
fix small grammar issues. 
Introduction Motivation: The Introduction quickly jumps into methodology. It might help to more
explicitly state the hypothesis (that encoding global structure in embeddings might align attention
with true behavior). 
Figure 1 (Architecture): The figure is clear . (No action needed beyond possibly improving resolution
if it appears blurry.) 
Result Presentation: In Table II, underline/bold formatting is defined but not visible in plain text;
ensure figures/tables are legible. Consider adding performance results for all relevant baselines
mentioned in text (e.g. include entries for SHAP or other explainers if evaluated).

[ ] Provide full methodological details: Report all model and training hyperparameters (embedding
size, number of layers/heads, optimizer , learning rate, epochs, random seeds, etc.) and dataset
splits. Justify and detail the Markov embedding dimension. 
[ ] Share code and data: Publish code (model, training, evaluation scripts) and, if possible, the
processed dataset or links to it. Include a “Code Availability” statement and deposit in a public
repository (e.g. GitHub/Zenodo) as per best practices. 
[ ] Expand evaluation: Run experiments on additional datasets (or provide more folds/seeds) to test
generality. If not feasible, add a detailed discussion of limitations.

Recommended Reviewer Expertise and Conflicts
Expertise: Potential reviewers should be experts in machine learning for cybersecurity (especially Android
malware detection) and explainable AI. Experience with sequence models (Transformers, RNNs) and
interpretability methods would be ideal. Knowledge of reproducibility practices is a plus. 
Conflicts to Monitor: The author is affiliated with [Institution Name / UrbanCompany]. Ensure reviewers
have no recent collaborations or institutional ties with the author . No obvious conflict is apparent. It would
be prudent to avoid reviewers from the same organization or recent co-authors of the author , and to
include diversity in the selection (academic and industry backgrounds).

evaluation. Specifically, we now list all hyperparameters (embedding dimensions, learning rates, etc.), training
settings, and data splits in the Methods section and include a link to our open-source code repository. We have
performed additional experiments (including [e.g. Drebin dataset]) and applied statistical tests (McNemar’s test) to
confirm the reported effects. We expanded the Discussion of related work, adding citations to recent explainable
malware studies and clarifying our novelty. Finally, we addressed notation issues (clarified the meaning of k) and
corrected formatting errors in the tables. These revisions should make the methodology and conclusions much
clearer and reproducible.
